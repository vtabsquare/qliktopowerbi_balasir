Understood. You are not asking only for an AI chat panel that explains errors. You want the embedded AI agent to behave like ChatGPT in a development environment:

Detect the error
→ understand the original Qlik logic
→ identify the real root cause
→ generate the corrected Power Query, DAX, model, or metadata
→ show the proposed change
→ apply the correction
→ rerun validation
→ continue fixing related errors until the object is valid

That requires a true AI correction workflow, not only an informational chatbot.

Required behavior

When an error appears in the application, provide a button such as:

Fix with AI

The AI agent must then perform the following sequence.

The agent must retrieve:

Current error message

Exact line and token

Original Qlik script
Upstream and downstream lineage
Available tables and fields
Current datatype registry

2. Diagnose the root cause

The AI must determine whether the error is caused by:

An incorrect query name
A missing source
A missing column
An incorrect step order
A stale generated query
A datatype mismatch
A broken join

An invalid DAX translation

A relationship problem
An unsupported Qlik construct

The agent should explain the actual root cause before proposing a correction.

Example:

Root cause

The original Qlik script defines RegionMap as a MAPPING LOAD, but the Power Query generator emitted a reference to #"RegionMap" without creating the corresponding lookup query.

This is a generation defect, not a source-data error.
3. Generate the smallest safe correction

The AI should not rewrite the entire query unless necessary.

It must create a structured patch.

Example:

interface AiCorrectionProposal {
    proposalId: string;
    projectId: string;
    projectVersion: string;

    targetType: "M" | "DAX" | "datatype" | "join" | "relationship";
    targetObject: string;

    rootCause: string;
    evidence: EvidenceReference[];

    originalCode: string;
    correctedCode: string;
    patchOperations: PatchOperation[];

    affectedObjects: string[];
    riskLevel: "low" | "medium" | "high" | "critical";
    confidence: number;

    requiredValidations: string[];
}

For a missing mapping query, the AI may propose:

RegionMap =
    Table.SelectColumns(
        Regions_Staging,
        {"RegionID", "RegionName"}
    )

Or it may determine that an inline merge is safer:

#"Merged RegionMap" =
    Table.NestedJoin(
        PreviousStep,
        {"RegionID"},
        Regions_Staging,
        {"RegionID"},
        "RegionLookup",
        JoinKind.LeftOuter
    ),

#"Expanded RegionMap" =
    Table.ExpandTableColumn(
        #"Merged RegionMap",
        "RegionLookup",
        {"RegionName"},
        {"RegionName"}
    ),

#"Applied Region Default" =
    Table.ReplaceValue(
        #"Expanded RegionMap",
        null,
        "Unknown Region",
        Replacer.ReplaceValue,
        {"RegionName"}
    )

The agent must choose based on the actual project lineage.

16. Important design rule

The preferred correction hierarchy must be:

Correct parser metadata
→ correct canonical graph
→ regenerate execution plan
→ regenerate M/DAX/model

Avoid this approach:

Directly modify generated M only

A direct M patch can be allowed for a manual user override, but the underlying metadata must be synchronized so the next regeneration does not remove the correction.
