Prompt: Enhance QlikView-to-Power BI Migration Application
Objective

Enhance the existing QlikView-to-Power BI migration application so that it accurately reconstructs the Qlik load-script logic, produces a clean Power Query and Power BI semantic model, and provides a reliable validation and correction workflow.

The implementation must be generic and work dynamically for different QlikView applications, QVS files, QVW/PRJ exports, source files, table names, field names, joins, composite keys, variables, expressions, and data models.

Do not hard-code fixes for Sales, Calendar, Rolling Sales, or any current test data.

The enhancement must address these core areas:

Validation issues must disappear immediately after they are fixed.
Composite keys must be created only when genuinely required.
Qlik joins must be reconstructed correctly in Power Query.
Duplicate attributes must be removed from the final semantic model.
Unnecessary tables and keys must not be exported.
The UI must clearly show what was fixed and where.
Every migration decision must be traceable and testable.

3. Reconstruct composite keys correctly
Requirement

When Qlik associates or joins two tables using multiple fields, create a Power BI composite key from those fields.

Example:

Sales:
CustomerID
RegionCode

CustomerRegion:
CustomerID
RegionCode

The generated key may be:

CompositeKey_CustomerID_RegionCode =
    Text.From([CustomerID], "en-US")
    & "¦"
    & Text.From([RegionCode], "en-US")

The same key expression must be generated in both tables.

Do not create unnecessary keys

A composite key must be generated only when all the following are true:

Two or more fields participate in the same logical association or join.
The same ordered field set exists on both sides.
A single stable key is not already available.
The fields are not merely unrelated columns that happen to share names.
The relationship is required in the final Power BI model.
The resulting key does not create a many-to-many ambiguity unnecessarily.

Do not create composite keys from every collection of common columns.

Composite-key signature

Use a deterministic signature:

function compositeKeySignature(fields: string[]): string {
  return fields
    .map(normalizeIdentifier)
    .sort()
    .join("|");
}

However, preserve the original join-field order in the generated concatenation when order has semantic significance.

Key deduplication

Before creating a key, check:

Does the same composite-key signature already exist?
Does an equivalent physical key already exist?
Is this key used by a relationship?
Is this only an intermediate Qlik association?
Will the table be excluded from the semantic model?

Do not generate:

Duplicate composite-key columns
Keys for staging-only tables
Keys for tables removed from the final model
Keys that are never used
Multiple keys representing the same field set
Both a single-field relationship and equivalent composite relationship
Null-safe key generation

Generate null-safe and collision-safe keys:

CompositeKey =
    Text.Combine(
        {
            if [CustomerID] = null
                then "<NULL>"
                else Text.From([CustomerID], "en-US"),

            if [RegionCode] = null
                then "<NULL>"
                else Text.From([RegionCode], "en-US")
        },
        "¦"
    )

Escape separator values when the source data can contain the delimiter.

4. Reconstruct Qlik JOIN logic in Power Query
Required behaviour

When the Qlik script contains:

LEFT JOIN (Sales)
LOAD
    CustomerID,
    CustomerName,
    CustomerGroup
RESIDENT Customers;

generate equivalent Power Query logic:

Joined_Customers =
    Table.NestedJoin(
        Sales_PreviousStep,
        {"CustomerID"},
        Customers_Staging,
        {"CustomerID"},
        "CustomersJoin",
        JoinKind.LeftOuter
    ),

Expanded_Customers =
    Table.ExpandTableColumn(
        Joined_Customers,
        "CustomersJoin",
        {"CustomerName", "CustomerGroup"},
        {"CustomerName", "CustomerGroup"}
    )

5. Avoid duplicate columns after joins

Before Table.ExpandTableColumn, calculate:

const mainColumns = getAvailableColumns(mainTable);
const joinKeys = getJoinKeys(joinOperation);
const requestedColumns = getQlikJoinOutputColumns(joinOperation);

const expandableColumns = requestedColumns.filter(column =>
  !joinKeys.includes(column) &&
  !mainColumns.includes(column)
);

When the column exists in both tables but has different business meaning, do not silently remove it. Apply a deterministic qualified name and flag it for review:

Customers.CustomerStatus
Sales.CustomerStatus

When it is the same business attribute, preserve only one final semantic column.

6. Prune duplicated attributes from secondary tables
User requirement

When attributes from a lookup or secondary table have already been joined into the main table, do not keep the same duplicated attributes in both final semantic-model tables unless they are genuinely needed.

Example:

CustomerID
CustomerName
CustomerGroup
Country
CustomerCategory

If Qlik joins these fields into Sales:

CustomerGroup

The model query should retain only fields needed for:

Business-required dimension analysis

9. DAX aggregation policy

Qlik script aggregations such as:

LOAD
    CustomerID,
    Sum(SalesAmount) AS TotalSales
RESIDENT Sales
GROUP BY CustomerID;

should generally become:

Total Sales =
SUM('Sales'[SalesAmount])

Do not materialise a SalesSummary Power Query table unless:

It changes grain required by later ETL logic.
It is joined to other datasets.
It is reused as an input to subsequent Qlik transformations.
Its result cannot be reproduced safely through DAX filter context.
Performance testing proves materialisation is required.

The migration decision must appear in the reconstruction report.
